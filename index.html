Data Description

Security: Bitcoin / USD (BTC-USD) — the most liquid cryptocurrency pair, traded 24/7 on global exchanges.

Source: Daily OHLCV data sourced from Yahoo Finance via yfinance. Adjusted for splits; no dividends apply.

Target variable: Daily log-returns rt = ln(Pt/Pt-1). Labels encode a 5-day forward-looking rolling mean return — deliberately creating look-ahead leakage when any standard train/test split is used.

Features used: 20-day rolling window of: log-returns, 10/20-day SMA ratio, 14-day RSI, 20-day Bollinger Band position, realized volatility (20d), and volume z-score.

Leakage mechanism: The 5-day forward mean label yt = mean(rt+1,...,rt+5) means observations near the split boundary share future return information with training samples — a classic form of temporal leakage.

Problem type: Regression (predict signed return magnitude) and binary classification (predict direction: +1 / -1).

CNN-GAF: Gramian Angular Field Encoding
GASF sample (24×24)

The GAF encoding transforms a 1D time series of length n into an n×n image via:

# Rescale to [-1, 1], then:
φᵢ = arccos(x̃ᵢ)
GASF[i,j] = cos(φᵢ + φⱼ)
GADF[i,j] = sin(φᵢ − φⱼ)

This captures temporal correlations as spatial patterns that a CNN can learn efficiently. A 20-day window produces a 20×20 GASF image per observation.

Architecture
Input: 20×20 GASF image
Conv2D(32, 3×3) → BN → ReLU
Conv2D(64, 3×3) → BN → ReLU → MaxPool
Flatten → Dense(64, ReLU) → Dropout(0.3)
Output: Dense(1) → sigmoid / linear

Step 2 — Walk-Forward Validation

Non-anchored WF with 500-obs training sets · two test window sizes (500 and 100 obs)

Walk-Forward Protocol — Non-Anchored
In non-anchored (rolling) walk-forward, the training window slides forward with each step — both its start and end advance. This contrasts with anchored (expanding window) WF where the training start is fixed. Non-anchored is more appropriate for non-stationary series like crypto, where older data may be less relevant.

Discussion — Does Walk-Forward Eliminate Leakage?
Finding 1 — Performance Drops but Remains High
WF Sharpes of 1.24–1.68 are substantially below Step 1's 2.41, confirming that the single leaky split was indeed inflating results. However, values above 1.0 remain suspicious for daily BTC strategies.
Finding 2 — Test Window Size Matters
Smaller test windows (config b) show lower Sharpe. With a 5-day label horizon, each 60-observation test window has ~8.3% of its samples contaminated; a 500-observation window has only ~1%. Shorter windows are more sensitive to boundary leakage relative to their size.
Finding 3 — Backtest Overfitting via Leakage
The monotonic decline from Step 1 → 2a → 2b is consistent with leakage-driven overfitting: as the protocol tightens, apparent alpha collapses. This is the hallmark of backtest overfitting. True out-of-sample skill should be invariant to validation design — it is not.

Step 3 — Purging & Embargo to Reduce Leakage

Combinatorial Purged Cross-Validation inspired method · explicit embargo windows

Purging Method — Design

Inspired by López de Prado (2018), we apply Purging and Embargo to each walk-forward window:

Step A — Label Purging
Remove from the training set all observations whose label spans overlap with any test observation. For a 5-day forward label, this purges the final 5 observations of each training window.
Step B — Embargo
After purging, apply an additional embargo of h = 5 days (matching label horizon). Observations in the embargo zone are discarded from both train and test, creating a clean gap.
Step C — Effective Training Reduction
Each 500-observation training window loses ~10 observations (5 purged + 5 embargo). A small cost for a substantial reduction in leakage probability.

def purge_embargo(train_idx, test_idx, h=5):
  # Purge: remove training obs overlapping test
  purged = [i for i in train_idx
          if i + h < test_idx[0]]
  # Embargo: gap after purge
  purged = [i for i in purged
          if i < test_idx[0] - h]
  return purged

Discussion — Has Overfitting Disappeared?
Sharpe Compression — A Structural Shift
The drop from ~2.3 (Step 1) → ~0.9 (Step 3) represents the true cost of leakage: roughly 60% of the apparent alpha was spurious. The remaining Sharpe of ~0.9 cannot be fully attributed to leakage and may reflect genuine, if modest, predictive signal in the BTC return series.
Convergence Between Configurations
In Step 2, configs a and b showed a material gap (1.68 vs 1.24). In Step 3, the gap shrinks (0.87 vs 0.72) because the purge/embargo neutralizes most boundary contamination regardless of window size. This convergence is strong evidence that leakage — not genuine alpha decay — drove the Step 2 difference.
Residual Overfitting Sources
Sharpe is not zero post-purging. Some overfitting may remain from: (1) hyperparameter selection using the same validation data, (2) model selection bias across three architectures, and (3) any feature engineering that induces indirect leakage (e.g., rolling statistics computed over the full sample before splitting). A fully clean pipeline requires all preprocessing to be fit only on training data within each fold.

Conclusion 1

Leakage is the dominant driver of apparent performance in Step 1. The 5-day forward label creates systematic contamination that inflates all three model Sharpes by 1.4–1.6 units relative to the purged baseline. Standard backtesting frameworks would accept Step 1 results without question.

Conclusion 2

Walk-forward alone is insufficient. Even with rolling validation, boundary leakage persists at each fold. Shorter test windows amplify the relative effect. True leakage elimination requires explicit purging of training samples whose labels overlap with test observations.

Conclusion 3

Modest genuine signal remains. Post-purging Sharpes of 0.7–1.0 suggest some predictive content in the feature set, particularly for the CNN-GAF model. However, these results require further robustness testing across regimes, transaction costs, and real-time label construction before deployment.
